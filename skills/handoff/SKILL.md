# Handoff — 会话交接 Skill

管理 AI Agent 之间的上下文交接，支持两种模式：**写入交接文档**（当前会话结束时）和**恢复交接文档**（新会话开始时）。

## 模式判断

根据触发方式自动判断模式：

| 触发方式 | 模式 |
|----------|------|
| 用户说"交接"、"handoff"、"生成交接文档" | **Write** |
| `/handoff` 命令 | **Write** |
| 用户说"继续上次"、"接上次"、"pick up" | **Resume** |
| 新会话开始，用户提及要继续之前的工作 | **Resume** |

---

## Write 模式 — 生成交接文档

### 1. 确定输出路径

运行脚本获取文件路径（自动处理重名）：

```bash
bash <SKILL_DIR>/scripts/write-handoff.sh "<工作目录>"
```

脚本输出带日期的文件路径，如 `260330-handoff.md`。同日多次交接自动追加序号（`260330-handoff-2.md`）。

### 2. 回顾当前会话

梳理整个会话历程，提取以下信息：

- 任务的起因和目标
- 已完成的具体工作（附 commit hash、PR 链接、文件路径）
- 关键技术决策及其理由
- 用户的明确偏好和约束
- 尚未解决的问题

### 3. 写入交接文档

按以下结构写入文件：

```markdown
# 交接文档 — <项目/任务简述>

生成时间: YYYY-MM-DD
工作目录: `<绝对路径>`

---

## 1. 当前任务目标
说明要解决的问题、预期产出和完成标准。

## 2. 当前进展
已完成的分析、修改、排查、讨论或产出。附具体的 commit、文件路径、命令。

## 3. 关键上下文
- 重要背景信息
- 用户的明确要求
- 已知约束
- 已做出的关键决定及理由
- 重要假设

## 4. 关键发现
最重要的结论、规律、异常点、根因判断或设计判断。

## 5. 未完成事项
按优先级列出仍需处理的内容。

## 6. 建议接手路径
- 应优先查看哪些文件/模块/日志
- 应先验证什么
- 推荐的下一步动作

## 7. 风险与注意事项
哪些点容易误判、重复劳动或跑偏。哪些方向已验证过且不建议继续。

## 下一位 Agent 的第一步建议
明确的第一步行动建议。
```

### 4. 写作标准

- 这是写给另一个 AI Agent 的文档，不是面向用户的总结
- 信息具体，引用文件路径、类名、接口名、命令
- 保留对继续工作最有价值的可执行信息
- 避免空话和客套话
- 使用中文撰写

### 5. 确认输出

写入后告知用户文件路径，并提示可在新会话中使用 Resume 模式恢复。

## Resume 模式 — 从交接文档恢复

### 1. 查找最新交接文档

运行脚本查找当前项目的最新 handoff 文件：

```bash
bash <SKILL_DIR>/scripts/find-handoff.sh "<工作目录>"
```

如果脚本返回 `NO_HANDOFF_FOUND`，告知用户当前目录无交接文档。

### 2. 读取交接文档

使用 Read 工具读取找到的 handoff 文件全文。

### 3. 验证关键状态

交接文档是历史快照，恢复前必须验证：

- **Git 状态**：运行 `git log --oneline -5` 和 `git status` 确认当前分支和代码状态
- **文件存在性**：交接文档中提到的关键文件是否仍然存在
- **未完成事项**：哪些已被后续会话完成，哪些仍然 pending

### 4. 生成恢复摘要

向用户报告：

```
已从 <文件名> 恢复上下文：
- 任务：<一句话概述>
- 进展：<已完成 N 项，剩余 M 项>
- 建议下一步：<具体行动>
```

### 5. 等待用户指令

恢复完成后等待用户确认方向，不要自动开始执行未完成事项。

## 文件命名与存储

| 项目类型 | 存储位置 |
|----------|----------|
| 有 `handoffs/` 目录 | 写入 `handoffs/` |
| 有 `docs/handoffs/` 目录 | 写入 `docs/handoffs/` |
| 其他 | 写入项目根目录 |

写入前检查项目是否已有 `handoffs/` 或 `docs/handoffs/` 目录。如果有，使用现有目录；如果没有，写入项目根目录。不要自作主张创建新的 handoffs 目录。
